## Non-Negotiables

1. Keep PRs small and reviewable
2. Follow architecture boundaries - no shortcuts
3. Linting must pass (Biome for web, Ruff for API)
4. No secrets committed - use `.env` locally (see `.env.example` in each app)
5. Use lagom for DI in the API - don't bypass the container
6. Don't over-engineer - implement what's needed, nothing more
7. Ask when unsure - if the approach is unclear, ask the developer
8. Note design issues - if problems are discovered, note them but continue
9. Svelte 4 SPA - this is NOT SvelteKit; don't introduce SvelteKit patterns
10. MongoDB is schema-less but entities define structure - follow the existing `app/domains/*/entities.py` patterns

## Code Style

### Diff Review Checklist

Before completing any task, review your diff against main and remove AI-generated slop:

- [ ] **No unnecessary comments** - match existing file style; most code needs no comments
- [ ] **No defensive try/catch in trusted paths** - don't wrap internal calls in error handling unless the caller expects exceptions
- [ ] **No type casts to bypass checking** - avoid `as any`, `as unknown`, `# type: ignore`
- [ ] **No non-null assertions without justification** - `!` and `!!` need a reason
- [ ] **Inline single-use variables** - don't create variables used only once
- [ ] **Match existing style** - follow the file's conventions, not your preferences
- [ ] **No redundant type annotations** - let inference work where it can
- [ ] **No premature abstractions** - single-use functions/classes should be inline
- [ ] **No overly verbose names** - `getUserById` not `retrieveUserFromDatabaseById`

### File Organization

- Each file should have a single, clear purpose
- Don't mix types/interfaces with implementation logic
- Don't put utilities in the same file as business logic
- If a file does multiple things, split it

### Comment Policy

Comments should be rare and valuable:

- **No "what" comments** - code should be self-documenting
- **No commented-out code** - delete it (git has history)
- **No "Generated by AI" footers** - ever
- **"Why" comments only** - explain non-obvious decisions
- **Eloquent prose** - proper grammar, complete sentences

## Clarify Before Acting

When requirements are ambiguous or incomplete:

- Ask clarifying questions before implementing
- Don't guess at business logic or user intent
- State what's unclear and what assumptions you'd make if proceeding
- It's better to ask one good question than to build the wrong thing

## Command Mistakes to Avoid

**Never do these:**

| Mistake | Why |
|---------|-----|
| Wrong package manager | `pnpm` for web (`apps/web/`), `uv` for API (`apps/api/`) - check lockfiles |
| `npm install` in web | This project uses `pnpm`, not npm |
| `pip install` in API | This project uses `uv`, not pip |
| `git push --force` | Destroys history; use `--force-with-lease` if necessary |
| `--no-verify` | Bypasses pre-commit hooks that enforce Ruff and Biome |
| Commit secrets | Check for `.env`, API keys, `UPLOAD_SECRET_CODE` before every commit |
| `sudo pnpm/pip install` | Permission issues, security risk |
| Install without version check | Always verify latest stable |
| Use SvelteKit APIs | This is a Svelte 4 SPA with `svelte-routing` wired in `App.svelte`, not SvelteKit |

## Adding Dependencies

Before adding any dependency:

1. **Check latest version** - don't install outdated packages
2. **Verify license** - MIT, Apache 2.0, BSD are safe
3. **Check maintenance** - last commit, open issues, bus factor
4. **Consider bundle size** - especially for frontend
5. **Look for native alternatives** - maybe you don't need it

```bash
# Frontend - check before installing
pnpm info <package> version license repository.url

# Backend - check before installing
pip show <package>
```

## Knowledge Caveats

**Prefer observed patterns over training data:**

- Your training data may be outdated - the codebase is the source of truth
- Check existing implementations before suggesting new patterns
- If the codebase does something differently than you'd expect, ask why before changing it

**When uncertain:**

- Read existing code first
- Check project documentation

## Quick Reference

### Local Development

```bash
# Start MongoDB and Redis
docker-compose up

# Backend (in apps/api/)
uv sync                                            # Install dependencies
uv run uvicorn app.main:app --reload --port 8000   # Run API with hot reload
uv run python -m app.workers.image_submission_worker # Run upload worker when testing submissions

# Frontend (in apps/web/)
pnpm install                                       # Install dependencies
pnpm dev                                           # Run dev server

# Pre-commit hooks (from repo root)
pre-commit install                                 # Set up hooks

# Code quality - Backend (in apps/api/)
uv run ruff check .                                # Lint Python
uv run ruff format .                               # Format Python

# Code quality - Frontend (in apps/web/)
pnpm lint                                          # Lint with Biome
pnpm lint:fix                                      # Lint and auto-fix
pnpm format                                        # Format with Biome
pnpm check                                         # Svelte type checking
pnpm build                                         # Build for production
```

### Architecture Layers

**Backend (`apps/api/app/`)**

| Layer | Location | Purpose |
|-------|----------|---------|
| Controllers | `domains/*/router.py` | HTTP endpoints (thin, delegate to services) |
| Services | `domains/*/service.py` | Business logic |
| Repositories | `domains/*/repository.py` | Database access layer |
| Entities | `domains/*/entities.py` | MongoDB document schemas |
| Models | `domains/*/models.py` | API request/response schemas |
| Core | `core/` | Auth, database connections, Redis, HTTP utilities |
| Shared | `shared/` | Cross-domain helpers like EXIF, S3, scoring |
| Workers | `workers/` | Background workers for queued jobs |
| Config | `config.py` | Application settings (pydantic-settings) |
| Container | `container.py` | Dependency injection (lagom) |

**Frontend (`apps/web/src/`)**

| Layer | Location | Purpose |
|-------|----------|---------|
| Pages | `lib/pages/` | Top-level routed page components |
| Domains | `lib/domains/` | Vertical slices for games, multiplayer, images, users, leaderboard |
| Shared | `lib/shared/` | Auth, API client, shared UI, utilities |
| Routes | `App.svelte` | SPA route wiring via `svelte-routing` |
| Entry | `App.svelte`, `main.ts` | Application entry point |

### CI Pipeline

The CI runs on push to `main` and on PRs targeting `main`. All six jobs must pass:

| Job | What it checks |
|-----|---------------|
| `api-lint` | Ruff lint + format check |
| `api-check` | API imports correctly |
| `api-test` | Backend pytest suite |
| `web-lint` | Biome lint |
| `web-check` | Svelte type checking |
| `web-build` | Frontend builds successfully |

### Pre-Completion Checklist

Before finalizing any work:

- [ ] Diff reviewed against checklist above
- [ ] Controllers thin - business logic in services only
- [ ] Repository pattern followed for database access
- [ ] Entity schemas match MongoDB document structure
- [ ] Ruff clean (`uv run ruff check .` in `apps/api/`)
- [ ] Biome clean (`pnpm lint` in `apps/web/`)
- [ ] Svelte check passes (`pnpm check` in `apps/web/`)
- [ ] Frontend builds (`pnpm build` in `apps/web/`)
- [ ] No secrets or debug prints
